§1 Watchlist — any bike, not just the MT-07

One config file drives the whole system. Each entry is a bike search plus optional filters — adding a new bike later is an edit, not a code change.

config.yamlwatchlist definition
watches:
  - name: "MT-07"
    query: "yamaha mt-07"
    aliases: ["mt07", "mt 07"]  # catches inconsistent titles
    max_price: null     # e.g. 6000 — null means no filter
    min_year: null     # e.g. 2018
    max_km: null       # e.g. 30000
    location: null     # e.g. "Porto" — modifiable per watch, not hardcoded

  - name: "Honda CB500F"
    query: "honda cb500f"
    location: "Lisboa"

Filters narrow what counts as a match in your digest/dashboard. Listings that fail a filter are still stored, so they remain useful market context for scoring the ones that pass. location is a config field like any other — leave it unset to search nationwide, or set it per watch.

§2 Sources — Portugal

Each marketplace gets its own scraper behind one common interface: fetch(query) → list[Listing]. A fourth site later is an isolated addition.

Primary
Standvirtual.pt

Largest PT marketplace. Listing data is often embedded as structured JSON — more reliable to parse than raw HTML.

Primary
OLX.pt

Large general classifieds site with strong motorcycle presence.

Secondary
CustoJusto.pt

Smaller reach, but occasionally surfaces listings the other two miss.

§4 Data model

Every listing, from every site, is normalized to the same record before it touches the database.

  • watch_name, source, external id — identity & dedup key
  • title, price, year, km, location, url, image_url
  • first_seen, last_seen, is_active — lifecycle tracking
  • price_history — every price change, timestamped, to catch reductions

§5 Scoring — self-calibrating, no thresholds to tune

Every listing is ranked only against other active listings for that same watch — an MT-07 is never compared against a CB500F.

  • Price percentile within the watch's currently active pool, loosely adjusted for year/km outliers
  • Year and mileage as secondary factors
  • Price-drop detection — a reduced price is a strong signal independent of percentile
Great deal Good Fair Overpriced

Config filters decide what counts as a match; scoring decides how good each match is — the two operate independently, per watch.

§6 Scheduling & hosting

GitHub Actions runs the pipeline daily; GitHub Pages serves the result.

  • Daily cron (e.g. 07:00 UTC): checkout → set up Python → install deps → run bikedeals/main.py → commit data/listings.db and docs/ → push
  • Repo is public — required for free GitHub Pages hosting. No sensitive data is stored: just public listing links and prices, no secrets, no email addresses
  • Dashboard will live at https://<username>.github.io/BikeDeals/

§7 Notifications

Out of scope for v1, by request — dashboard is the interface for now.

notify.py is stubbed into the architecture on purpose, so email (or Telegram, etc.) slots in later without restructuring anything upstream of it.

§8 Scraping etiquette

Low, predictable load: a handful of searches, once a day, with realistic delays and a normal user-agent. Personal-use monitoring of public listing pages, not bulk harvesting.
