§1 The platform, not a config file

You add and edit watches through a real UI. No YAML, no repo access needed day-to-day. This is the biggest structural change from the first draft — it turns the project from "a script with a config file" into "an app with a database behind it."

What this costs

A static GitHub Pages site can't accept form input or persist your watches — it only serves files. A real UI needs a small always-on backend with a database behind it. Recommended shape:

  • Backend: a small API (FastAPI) storing watches, listings, and scores in SQLite/Postgres
  • Frontend: the form and table above, served by that same app — no separate service to keep in sync
  • Scraping: still runs once a day; either the app schedules it internally, or GitHub Actions pings a /run endpoint on a timer to guarantee it fires even if the free-tier host has gone to sleep
  • Hosting: a free-tier PaaS (Render, Railway, or Fly.io) instead of GitHub Pages, since Pages can't run a backend — see §7 for the open decision

§2 Filters

Every watch is defined by these fields, all optional except brand and model:

  • Brand — e.g. Yamaha
  • Model — e.g. MT-07
  • Price — min / max, in €
  • Year — min / max
  • Km — min / max
  • Location — optional; unset searches nationwide

Listings that fail a filter are still stored in the background — they remain useful market context for scoring the ones that pass, but won't clutter your view of active matches.

§3 Sources — Portugal

Each marketplace gets its own scraper behind one common interface: fetch(filters) → list[Listing]. A new source is an isolated addition, not a rewrite.

Primary
Standvirtual.pt

Largest PT marketplace. Listing data is often embedded as structured JSON — more reliable to parse than raw HTML.

Primary
OLX.pt

Large general classifieds site with strong motorcycle presence.

Secondary
CustoJusto.pt

Smaller reach, but occasionally surfaces listings the other two miss.

Secondary
Auto.pt

Adding as requested. To confirm during build: its motorcycle ("motos") catalog coverage looks thinner than cars — will verify listing volume before relying on it.

Best-effort
Facebook Marketplace

Real caveat: FB has no public search API, requires a logged-in session, aggressively blocks automated access, and its ToS prohibits scraping. Feasible as an opt-in, low-frequency, best-effort source using your own session — but expect it to break more often than the others, and there's a small risk of the account being flagged. Recommend shipping without it first, adding it later if the other four aren't enough.

§5 Data model

Two tables now instead of one flat list: what you're watching, and what's been found.

  • Watch — brand, model, price/year/km min–max, location, active sources
  • Listing — watch_id, source, external id, title, price, year, km, location, url, image_url, first_seen, last_seen, is_active
  • PriceHistory — every price change per listing, timestamped, to catch reductions

§6 Scoring — self-calibrating, no thresholds to tune

Every listing is ranked only against other active listings for that same watch — an MT-07 is never compared against a CB500F.

  • Price percentile within the watch's currently active pool, loosely adjusted for year/km outliers
  • Year and mileage as secondary factors
  • Price-drop detection — a reduced price is a strong signal independent of percentile
Great deal Good Fair Overpriced

Your filters decide what counts as a match; scoring decides how good each match is — the two operate independently, per watch.

§7 Hosting — open decision

GitHub Pages served the old static-site design for free, but it can't run a backend or accept form submissions. The platform pivot means picking a new host.

  • Render / Railway / Fly.io free tier — runs the FastAPI app, its database, and the scheduled scrape together in one place. Free-tier apps typically sleep after inactivity, so a daily GitHub Actions ping to a /run endpoint keeps the schedule reliable without paying for an always-on instance.
  • Repo can stay public or go private now — Pages is no longer the reason to keep it public, since the live app isn't served from Pages. Worth revisiting once the host is picked.

Flagging this rather than deciding it for you — which host to use affects setup steps below.

§8 Notifications

Still out of scope for v1, by request — the dashboard is the interface for now.

notify.py stays stubbed into the architecture on purpose, so email (or Telegram, etc.) slots in later without restructuring anything upstream of it.

§9 Scraping etiquette

Low, predictable load: a handful of searches per watch, once a day, with realistic delays and a normal user-agent. Personal-use monitoring of public listing pages, not bulk harvesting — and the one source that carries real ToS risk (Facebook) is opt-in and off by default.
